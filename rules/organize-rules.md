# 整理规则（rules/organize-rules.md）

整理知识库（organize）模式消化 inbox 草稿、生成报告、落盘时遵循的规则。

## 一、整理判断（对每张草稿）

### 合并判断

多张草稿指向**同一主题/同一认知单元** → 建议合并成一张正式卡片。

信号：
- 草稿 A 和 B 讲的是同一件事的两半（如「踩坑」+「修复」= 一次完整 fix）
- 草稿 A 和 B 是同一决策的不同侧面（如「架构调研」+「三族分类」= 一张架构卡）

合并后在报告里以「草稿A + 草稿B → 目标」的形式呈现，让用户一眼看到结构。

### 丢弃判断（归档，不真删）

以下草稿建议归档到 `90.archive/discarded/`：

- 寒暄、闲聊
- 未导致结论的探索
- 操作性指令来回（无知识价值）
- 与现有知识重复（已被别处更好覆盖）

**丢弃必须给理由**，报告里每条归档项附一句话原因——这是最该让用户过目的（防误杀）。

### 去向判断

按 `vaults/<vault>.md` 的**整理决策树**判断（不在这里重复，去对应 vault 适配器查）。本规则只管"要不要整理、要不要合并、要不要丢弃"，去向交给 vault 适配器。

## 二、对话循环铁律（三条，不可违反）

整理模式是**评审**，用户是最终决策者。三条铁律：

### 铁律 1：OK 前绝不动任何文件

报告生成后，**所有判断都是建议**。用户没明确说「OK / 执行 / 没问题」前，agent **绝不**移动、修改、删除任何 vault 文件。即使用户沉默也不算同意。

### 铁律 2：每次修订后重新生成完整报告

用户提异议后（如「第 3 张该去 60」「A 别合并」「别丢这条」），agent 修订判断表，然后**重新生成完整的 HTML 报告**（不是局部改）。保证用户看到的永远是自洽的全局图，不会因为局部改动而漏看影响。

### 铁律 3：agent 只改判断、不辩解

用户提异议时，agent **按用户的意志修订**，不反驳、不解释为什么自己原来的判断更好。用户的判断永远优先。agent 的角色是执行者，不是评审对手。

> 如果 agent 真的认为用户判断有风险，可以在修订后的报告里**用标注提示**（如「⚠️ 已按你要求改去向，但注意 X」），但仍以用户决定为准。提示是服务，不是争辩。

## 三、写入白名单（与 vault 适配器协同的硬护栏）

整理模式落盘时，agent 只能写入 **vault 适配器声明的白名单路径**。任何尝试写白名单外的操作，agent **直接拒绝并报错**。

对 Obsidian-KP：白名单 = `30.knowledge/` 内（详见 `vaults/obsidian-knowledge-palace.md` 三、硬约束）。绝碰轨道 A（ME.md/identity/skills/memory-stream/maps/context）。

## 四、落盘原子性

落盘分两阶段：
1. **staging**：所有目标文件先写进 `30.knowledge/.staging/`。
2. **commit**：全部 staging 成功后，原子移入正式房间。

中途任何失败 → staging 残留、正式区零污染。失败后报告错误，等用户指示。

## 五、清理

落盘完成后：
- 入库/合并的草稿：原 inbox 文件标 `status: promoted`，移到 `10.capture/promoted/`。
- 归档的草稿：移到 `90.archive/discarded/YYYY-MM/`，加 `status: discarded` + 丢弃理由。
- inbox 只剩**未评审**的草稿。
